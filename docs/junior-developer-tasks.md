## Задание 1. Кнопка «Наверх» (scroll-to-top)

**Цель:** При скролле вниз на любой публичной странице появляется кнопка, которая плавно возвращает пользователя к началу страницы.

### Что нужно сделать

- Создать React-компонент (например, `ScrollToTopButton`)
- Кнопка скрыта по умолчанию и появляется, когда пользователь прокрутил страницу вниз (примерно на 400-500 пикселей)
- По клику — плавный скролл наверх
- Кнопка должна быть фиксированной в правом нижнем углу экрана
- Анимация появления и исчезания (через CSS transitions)

### Где искать

- **Layout публичных страниц:** `platform/frontend/src/core/layouts/PublicLayout.tsx` — сюда нужно будет встроить компонент
- **Существующие публичные компоненты:** `platform/frontend/src/public/components/` — посмотри, как устроены другие компоненты, какой стиль кода используется
- **Стили:** `platform/frontend/src/styles.css` — все стили проекта в одном файле, ищи секцию `.public-layout` (примерно строка 322) для понимания структуры
- **Пример работы со скроллом:** `platform/frontend/src/public/hooks/usePublicRouteSwipe.ts` — здесь уже используется `window.scrollTo`, можно подсмотреть паттерн
- **i18n:** `platform/frontend/src/shared/i18n/en.ts` и `ru.ts` — если у кнопки есть `aria-label`, добавь перевод сюда

### Что понадобится знать

- `useState` — для хранения состояния "видна/скрыта"
- `useEffect` — для подписки на событие `scroll` при монтировании и отписки при размонтировании
- `window.scrollTo({ top: 0, behavior: 'smooth' })` — плавный скролл
- CSS `position: fixed`, `opacity`, `transition`

### Чего НЕ делать

- **Не ставить сторонних библиотек** — задача решается стандартными средствами React + CSS, никаких npm-пакетов
- **Не использовать `setInterval` или `requestAnimationFrame` для отслеживания скролла** — достаточно обычного `addEventListener('scroll', ...)`
- **Не добавлять кнопку в каждую страницу отдельно** — она должна быть в layout один раз
- **Не забывать убирать слушатель** — если не вызвать `removeEventListener` в cleanup-функции `useEffect`, будет утечка памяти
- **Не менять существующие компоненты** — только добавлять новый компонент и встраивать его в layout
- **Не делать кнопку слишком большой или яркой** — она не должна мешать контенту, посмотри общий стиль сайта (glass-эффекты, мягкие тени)

## Задание 2. Редактируемая дата публикации

**Цель:** В админке при редактировании проекта/поста добавить поле для выбора даты публикации вместо автоматической установки текущей даты.

### Что нужно сделать

- Добавить поле date picker в форму редактирования проекта в админке
- При создании нового поста — дата по умолчанию "сейчас", но можно изменить
- При редактировании существующего — показывать текущую дату публикации
- Дата должна отправляться на API при сохранении

### Где искать

- **Форма в админке:** `platform/frontend/src/core/pages/AdminProjectsWorkspace.tsx` — главный файл, здесь форма создания/редактирования. Обрати внимание на интерфейс `DraftProject` (начало файла) — в нем сейчас НЕТ поля `publishedAt`, его нужно добавить
- **Отправка данных на API:** `platform/frontend/src/public/data/project-store.ts` — ищи функцию `toApiPayload` и `ensurePublishedAt`. Сейчас `ensurePublishedAt()` автоматически ставит текущую дату при создании — логику нужно изменить, чтобы использовать дату из формы
- **Типы данных фронтенда:** `platform/frontend/src/public/types.ts` — тип `ProjectPost`, у него уже есть `publishedAt?: string`
- **Бэкенд (только для понимания, менять НЕ нужно):**
  - `platform/backend/src/Modules/ProjectPosts/Contracts/ProjectPostDtos.cs` — DTO уже принимает `PublishedAt` (тип `DateTimeOffset?`)
  - `platform/backend/src/Modules/ProjectPosts/Contracts/ProjectPostMappings.cs` — маппинг уже передает дату
  - API-эндпоинты (POST и PUT) уже принимают это поле — бэкенд трогать не нужно
- **i18n:** `en.ts` и `ru.ts` — добавить перевод для label поля ("Publication date" / "Дата публикации")

### Что понадобится знать

- HTML `<input type="datetime-local">` — нативный date picker браузера
- Преобразование дат: `new Date().toISOString()` и формат для `datetime-local` input
- Как работает state формы в этом проекте (посмотри, как связаны другие поля: `draft.title`, `draft.slug` и т.д.)

### Чего НЕ делать

- **Не трогать бэкенд** — API уже полностью поддерживает `PublishedAt`, менять нужно ТОЛЬКО фронтенд
- **Не ставить сторонние date picker библиотеки** (react-datepicker, MUI и т.д.) — используй нативный `<input type="datetime-local">`, он работает во всех современных браузерах
- **Не удалять логику `ensurePublishedAt`** — измени её так, чтобы она использовала дату из формы, а если форма пустая — подставляла текущую дату (fallback)
- **Не ломать существующие посты** — при загрузке существующего поста для редактирования дата должна корректно отображаться в поле
- **Не путать форматы дат** — `datetime-local` input ожидает формат `YYYY-MM-DDTHH:mm`, а API отправляет/принимает ISO 8601 (`2026-04-04T15:30:00Z`). Нужно конвертировать между ними
- **Не делать дату обязательной** — поле может быть пустым (для черновиков)

## Задание 3. Кнопка «Поделиться» для проектов

**Цель:** На странице детального просмотра проекта добавить кнопку «Поделиться» с нативным Web Share API и fallback меню для десктопа.

### Контекст

В проекте УЖЕ есть кнопка «Поделиться» для постов — компонент `PostActions`. Но она работает только для постов (`mode === "post"`), а для проектов её нет. Кроме того, текущая реализация не использует нативный Web Share API (который на мобильных устройствах открывает системное меню "Поделиться").

### Что нужно сделать

- На странице проекта (не поста) добавить кнопку «Поделиться»
- На мобильных — использовать `navigator.share()` (нативное системное меню)
- На десктопе (где `navigator.share` не поддерживается) — показывать выпадающее меню со ссылками (как уже сделано в `PostActions`)
- Кнопка «Копировать ссылку» должна быть в обоих случаях

### Где искать

- **Существующий компонент:** `platform/frontend/src/public/components/PostActions.tsx` — изучи его полностью, здесь уже реализовано: кнопка шаринга, выпадающее меню с соцсетями (VK, Telegram, Twitter, LinkedIn), копирование ссылки
- **Страница проекта:** `platform/frontend/src/public/pages/ProjectDetailPage.tsx` — ищи место, где для постов рендерится `PostActions` (примерно строка 337). Для проектов этот компонент не рендерится
- **Шапка детальной страницы:** `platform/frontend/src/public/components/ProjectDetailHeader.tsx` — здесь `detail-header__actions-row`, куда встраиваются кнопки действий
- **Стили:** в `styles.css` ищи секцию `.post-actions` — все стили для шаринга уже написаны
- **i18n:** `en.ts` и `ru.ts` — проверь, есть ли переводы для кнопки шаринга, при необходимости добавь

### Что понадобится знать

- `navigator.share({ title, url })` — Web Share API, возвращает Promise
- `navigator.canShare` или `'share' in navigator` — проверка поддержки
- `navigator.clipboard.writeText(url)` — копирование в буфер обмена
- Как переиспользовать существующий компонент или расширить его

### Чего НЕ делать

- **Не дублировать код** — не копируй весь `PostActions` и не создавай `ProjectActions`. Лучше переименуй/расширь существующий компонент, чтобы он работал и для постов, и для проектов. Или выдели общую логику
- **Не ломать существующий шаринг постов** — после твоих изменений кнопка на постах должна работать как раньше
- **Не ставить сторонних библиотек** — Web Share API, Clipboard API и ссылки на соцсети работают без пакетов
- **Не забывать про обработку ошибок** — `navigator.share()` может быть отклонён пользователем (AbortError), это не ошибка, не нужно показывать alert
- **Не показывать `navigator.share()` на десктопе** — сначала проверяй поддержку, иначе будет ошибка в консоли
- **Не хардкодить URL** — используй `window.location.href` или формируй URL из данных проекта

## Общие правила работы с проектом

### Как запустить

```bash
# Клонировать и запустить dev-окружение
docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build
```

Фронтенд будет доступен на `http://localhost` (через nginx).  
Админка: `http://localhost/app` (логин: `admin` / `admin123`).

### Структура проекта

```
platform/
  frontend/src/
    core/         — админка, авторизация, роутинг, layout
    public/       — публичные страницы и компоненты
    shared/       — общие утилиты, i18n
    styles.css    — все стили
  backend/src/    — ASP.NET Core бэкенд (НЕ трогать без необходимости)
  infra/          — nginx, postgres, Docker
```

### Как делать PR

1. Создай ветку от `main`: `git checkout -b feature/scroll-to-top`
2. Делай коммиты с осмысленными сообщениями на английском
3. Перед пушем проверь: `npm run typecheck --workspace @platform/frontend`
4. Создай Pull Request в `main` через GitHub
5. Дождись прохождения CI (сборка бэкенда + фронтенда)

### Общие запреты

- Не устанавливай npm-пакеты без согласования
- Не меняй файлы Docker, nginx, CI/CD pipeline
- Не трогай авторизацию и безопасность
- Не меняй существующие стили других компонентов
- Не коммить файлы `.env`, секреты, бинарные файлы
